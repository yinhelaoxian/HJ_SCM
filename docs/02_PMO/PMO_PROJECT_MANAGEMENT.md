**文档版本**: v1.0  
**生效日期**: 2026-02-13  
**适用范围**: 多Agent协作项目  
**保密等级**: 内部

## 一、总则

### 1.1 目的

规范多Agent协作项目管理流程，确保虚拟团队高效交付，沉淀可复用的协作范式和最佳实践。

### 1.2 核心原则

| 原则 | 说明 |
|------|------|
| **角色分工** | 9大Agent各司其职 |
| **并行协作** | 多Agent同步推进 |
| **角色自治** | Agent自主决策 |
| **经验沉淀** | 文档驱动协作 |

### 1.3 适用场景

本制度适用于SCM数字化转型项目及其衍生项目。

## 三、项目生命周期

### 3.1 阶段划分

```
启动 → 规划 → 设计 → 开发 → 测试 → 上线 → 复盘
     →  验收报告 →  结项报告
```

### 3.2 Agent协作流

```
启动阶段
    │
    ├── 产品经理 → 需求文档
    ├── 业务专家 → 业务背景
    └── 系统架构师 → 技术选型

设计阶段
    │
    ├── 系统架构师 → 架构设计
    ├── 高级工程师 → 接口定义
    └── 业务专家 → 方案评审

开发阶段
    │
    ├── 高级工程师 → 功能开发
    ├── OpenCode官 → 代码规范
    └── 超级助手 → AI增强

测试阶段
    │
    └── QA Tester → 测试用例

上线阶段
    │
    ├── Docker专家 → 容器化
    └── 交付官 → 发布清单

复盘阶段
    │
    ├── PMO → 经验沉淀
    └── 业务专家 → 改进建议
```

## 七、变更管理与代码提交制度

### 7.1 代码提交规范

#### 7.1.1 提交时机

**立即提交原则**：
- ✅ 完成任何功能修复后**立即提交**
- ✅ 完成任何样式调整后**立即提交**
- ✅ 完成任何文档更新后**立即提交**
- ✅ 解决任何Bug后**立即提交**
- ✅ 任何可运行、可验证的变更**立即提交**

**禁止行为**：
- ❌ 禁止积累大量未提交变更（超过1天）
- ❌ 禁止在一天结束时批量提交
- ❌ 禁止在多天后才提交

#### 7.1.2 提交信息规范

```markdown
<类型>: <简短描述>

[可选的详细说明]

[关联的Issue或任务编号]

类型标识：
- fix: 修复Bug
- feat: 新功能
- docs: 文档更新
- style: 样式调整
- refactor: 重构代码
- perf: 性能优化
- test: 测试相关
- chore: 构建/工具/依赖更新
```

**示例**：
```
fix: 修复首页布局错位问题

- 修正侧边栏宽度计算
- 更新Tailwind CSS v4兼容性
- 修复@import语法顺序错误

Closes #156
```

#### 7.1.3 提交前检查清单

| 检查项 | 负责人 | 通过标准 |
|--------|--------|----------|
| 代码编译通过 | 高级工程师 | `npm run build` 成功 |
| 语法错误清除 | 高级工程师 | 无编译警告 |
| 样式检查通过 | 高级工程师 | CSS无错误 |
| 单元测试通过 | QA Tester | 测试覆盖率≥80% |
| 文档同步更新 | 业务专家 | 关键变更已记录 |

### 7.2 Git分支策略

#### 7.2.1 分支命名规范

| 分支类型 | 命名格式 | 说明 |
|----------|----------|------|
| 主分支 | `main` | 生产环境代码 |
| 开发分支 | `develop` | 开发主分支 |
| 功能分支 | `feature/<issue>-<描述>` | 新功能开发 |
| 修复分支 | `fix/<issue>-<描述>` | Bug修复 |
| 发布分支 | `release/<版本号>` | 版本发布准备 |
| 文档分支 | `docs/<描述>` | 文档更新 |

#### 7.2.2 分支管理流程

```
main (生产环境)
    ↑
    ├── develop (开发主分支)
    │       ↑
    │       ├── feature/123-add-dashboard
    │       ├── feature/456-fix-sidebar
    │       └── feature/789-update-docs
    │
    └── release/v1.0.0 (发布分支)
```

### 7.3 代码Review制度

#### 7.3.1 Review标准

| 代码类型 | Review要求 | Reviewer |
|----------|-----------|----------|
| 核心业务逻辑 | 强制Review | 高级工程师/架构师 |
| UI/样式调整 | 快速Review | 高级工程师 |
| Bug修复 | 强制Review | 高级工程师 |
| 文档更新 | 自审即可 | 提交者自审 |
| 配置文件 | 快速Review | OpenCode官 |

#### 7.3.2 Review检查项

- [ ] 代码逻辑正确性
- [ ] 命名规范符合
- [ ] 注释清晰完整
- [ ] 无硬编码数值
- [ ] 异常处理完善
- [ ] 性能无明显问题
- [ ] 测试覆盖充分

### 7.4 版本发布规范

#### 7.4.1 版本号规则

采用语义化版本：`主版本.次版本.修订号`

- **主版本 (MAJOR)**: 不兼容的API变更
- **次版本 (MINOR)**: 向后兼容的功能新增
- **修订号 (PATCH)**: 向后兼容的缺陷修复

#### 7.4.2 发布检查清单

| 检查项 | 执行人 | 标准 |
|--------|--------|------|
| 所有功能测试通过 | QA Tester | 测试通过率100% |
| 性能基准测试 | 高级工程师 | 无性能退化 |
| 安全扫描通过 | Docker专家 | 无高危漏洞 |
| 文档更新完成 | 业务专家 | 关键文档同步 |
| 回滚方案验证 | Docker专家 | 回滚脚本可用 |
